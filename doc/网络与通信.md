# 网络与通信

## 概述

大博弈采用客户端-服务端架构，通过 WebSocket 进行双向通信。服务端为游戏的唯一权威来源，客户端为无状态的渲染终端。网络层使用非对称序列化策略以平衡可读性与性能。

## 通信协议

### WebSocket 连接

- **地址**：`ws://127.0.0.1:8080/ws`
- **协议**：标准 WebSocket（RFC 6455）
- **最大消息体**：256 MB（客户端配置）
- **最大帧大小**：64 MB（客户端配置）

### 消息方向与格式

| 方向 | 格式 | 帧类型 | 原因 |
|------|------|--------|------|
| 客户端 → 服务端 | JSON | Text | 命令消息体积小（< 1KB），JSON 可读性利于调试 |
| 服务端 → 客户端 | bincode | Binary | 状态快照体积大（~5MB），bincode 序列化快 5-10 倍 |

## 消息定义

### 客户端消息（ClientMsg）

```rust
// shared/src/lib.rs

pub enum ClientMsg {
    Tick,                    // 请求推进一个 tick
    IssueOrder(Order),       // 下达玩家指令
}

pub struct Order {
    pub kind: String,            // 指令类型
    pub target_province: Option<u32>,  // 目标省份
}
```

| 消息 | 触发时机 | 说明 |
|------|----------|------|
| `Tick` | GameSpeed 计时器到期且未暂停 | 请求服务端推进一天并返回状态 |
| `IssueOrder` | 玩家操作（当前未实现） | 排队到下一个 tick 执行 |

### 服务端消息（ServerMsg）

```rust
pub enum ServerMsg {
    StateSnapshot(GameState),  // 完整游戏状态快照
    Ack,                       // 指令确认
}
```

| 消息 | 返回场景 | 说明 |
|------|----------|------|
| `StateSnapshot` | 响应 `Tick` | 包含完整的 GameState（~50K 省份数据） |
| `Ack` | 响应 `IssueOrder` | 确认指令已入队 |

### 序列化兼容性

`ClientMsg` 和 `ServerMsg` 使用 serde 的默认外部标签格式（externally tagged），同时兼容 JSON 和 bincode：

```json
// JSON 格式示例（客户端 → 服务端）
"Tick"
{"IssueOrder": {"kind": "build", "target_province": 42}}
```

**设计决策**：最初使用 `#[serde(tag = "type", content = "payload")]` 邻接标签格式，但 bincode 不支持 `deserialize_identifier`，因此改为默认格式。

## 服务端实现

### WebSocket 处理器（`server/src/ws.rs`）

```
客户端连接 → actix_ws::handle() → 异步消息循环
```

#### 消息处理流程

```
接收 Text(json)
  ├── 解析为 ClientMsg
  ├── handle_msg() 处理
  │   ├── Tick:
  │   │   1. 排空 command_queue
  │   │   2. apply_commands()
  │   │   3. advance()（生产→消费→增长）
  │   │   4. 每 30 tick 持久化到 RocksDB
  │   │   5. 返回 StateSnapshot(gs.clone())
  │   └── IssueOrder:
  │       1. 推入 command_queue
  │       2. 返回 Ack
  ├── bincode::serialize(&reply)
  └── session.binary(bytes) 发送
```

### 持久化策略

- 不是每个 tick 都写入 RocksDB，而是**每 30 tick 写入一次**。
- 以 `"game_state"` 为键，JSON 序列化后存储。
- 服务端启动时从 RocksDB 恢复状态，实现断点续玩。

### 共享状态

```rust
// server/src/main.rs

pub struct AppState {
    pub game_state: Mutex<GameState>,
    pub command_queue: Mutex<Vec<Order>>,
    pub db: Mutex<GameDb>,
}
```

使用 `std::Mutex`（非 tokio::Mutex），因为锁持有时间极短且只有一个 WebSocket 连接。

## 客户端实现

### 网络插件（`client/src/net.rs`）

#### 线程模型

```
┌──────────────────────┐     ┌──────────────────────────┐
│  Bevy 主线程          │     │  网络 I/O 线程            │
│                      │     │  (独立 OS 线程 + tokio)    │
│  tick_sender ──────────────→│                          │
│         (CmdSender)  │     │  ws_loop():              │
│                      │     │    cmd_rx → sink.send()   │
│  state_receiver ←──────────│    stream.next() → state_tx│
│      (StateReceiver) │     │                          │
└──────────────────────┘     └──────────────────────────┘
```

网络 I/O 运行在独立的 OS 线程上，拥有自己的 tokio runtime。这避免了 WebSocket 的异步操作阻塞 Bevy 的游戏循环。

#### 通道设计

| 通道 | 类型 | 方向 | 容量 |
|------|------|------|------|
| `CmdSender` | `tokio::mpsc::UnboundedSender<ClientMsg>` | Bevy → WS | 无界 |
| `StateReceiver` | `std::sync::mpsc::SyncSender<GameState>` | WS → Bevy | 有界(8) |

- 命令通道无界：命令体积小且频率低，不会积压。
- 状态通道有界(8)：防止服务端响应积压导致内存暴涨；旧状态被丢弃，始终使用最新状态。
- `StateReceiver` 使用 `Mutex` 包装（`std::sync::mpsc::Receiver` 是 Send 但非 Sync）。

### 游戏速度与暂停

#### 资源

```rust
pub struct GameSpeed(pub Timer);  // 默认 1 秒/tick
pub struct Paused(pub bool);       // 默认 true（启动时暂停）
```

#### 系统

| 系统 | 说明 |
|------|------|
| `toggle_pause` | 空格键切换暂停状态 |
| `tick_sender` | 计时器到期时发送 Tick（暂停时跳过） |
| `state_receiver` | 每帧排空接收通道，保留最新状态 |

#### tick 发送逻辑

```rust
fn tick_sender(time, speed, sender, paused) {
    if paused.0 { return; }       // 暂停时不发送
    if speed.0.tick(delta).just_finished() {
        sender.send(Tick);         // 计时器到期，发送 Tick
    }
}
```

### 状态接收逻辑

```rust
fn state_receiver(receiver, latest) {
    while let Ok(state) = rx.try_recv() {
        latest.0 = Some(state);    // 非阻塞排空，保留最新
    }
}
```

使用 `try_recv()` 非阻塞读取，避免在无新状态时阻塞 Bevy 主线程。循环读取确保丢弃旧状态，始终使用最新快照。

## 网络性能

### 优化前后对比

| 指标 | JSON（优化前） | bincode（优化后） |
|------|--------------|-----------------|
| 状态快照大小 | ~37 MB | ~5 MB |
| 序列化时间 | 高 | 低（约 5-10 倍提升） |
| 反序列化时间 | 高 | 低 |
| 帧类型 | Text | Binary |

### 当前局限

- **全量快照**：每个 tick 发送完整的 GameState，包含所有 ~50K 省份。未来可考虑增量更新（delta encoding）只传输变化的省份。
- **单连接**：当前仅支持单玩家。多人模式需要连接管理和状态同步策略。
- **无压缩**：bincode 输出未做额外压缩。如果加入 zstd 或 lz4 压缩，可进一步减小传输体积。
- **无心跳/重连**：WebSocket 断开后不会自动重连。
