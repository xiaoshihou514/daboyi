# 架构总览

## 项目简介

大博弈（daboyi）是一款以 Rust 编写的大战略游戏。采用客户端-服务器架构，服务端负责游戏逻辑仿真与持久化，客户端负责地图渲染与用户交互，二者通过 WebSocket 通信。

## 技术栈

| 层级 | 技术选型 | 说明 |
|------|----------|------|
| 服务端框架 | actix-web 4 | 异步 HTTP/WebSocket 服务器 |
| 持久化 | RocksDB 0.24 | 嵌入式键值数据库 |
| 客户端引擎 | Bevy 0.15 | Rust 原生 ECS 游戏引擎 |
| 共享类型 | shared crate | 客户端与服务端共用的数据结构 |
| 地图编译 | mapgen 工具 | GeoJSON → bincode 离线处理 |
| 序列化 | serde + bincode/JSON | 命令用 JSON，状态快照用 bincode |
| 编译工具链 | clang + mold | clang 编译 RocksDB，mold 加速链接 |

## 核心设计原则

### 1. 服务端权威架构

所有游戏逻辑在服务端执行。客户端是无状态的——每个 tick 接收完整的 `GameState` 快照后重新渲染。客户端不做任何游戏逻辑计算，仅负责显示与用户输入。

### 2. 客户端驱动的 tick 模型

客户端按 `GameSpeed` 计时器的频率（默认每秒一次）向服务端发送 `Tick` 消息。服务端收到后推进一天的仿真，将完整状态返回。玩家可按空格键暂停/恢复。

### 3. 非对称序列化

- **客户端 → 服务端**：JSON 文本帧。命令消息（`Tick`、`IssueOrder`）体积极小，JSON 的可读性优先于性能。
- **服务端 → 客户端**：bincode 二进制帧。约 5 万省份的完整状态快照在 JSON 下约 37MB，bincode 压缩至约 5MB 且序列化速度提升 5-10 倍。

### 4. 插件化客户端

客户端使用 Bevy 的 Plugin 机制将网络（NetPlugin）、地图（MapPlugin）、UI（UiPlugin）三大子系统完全解耦。新增功能（如外交、军事）只需添加新插件。

### 5. 离线地图编译

地理空间数据的重处理（解析 GeoJSON、多边形简化、三角剖分）在 mapgen 工具中离线执行，输出 `map.bin`。运行时直接加载预处理后的二进制数据，避免启动时的计算开销。

## 数据流

```
1. 启动时：
   mapgen 工具 ──(GeoJSON)──→ assets/map.bin
   服务端加载 map.bin ──→ generate_world() ──→ GameState
   客户端加载 map.bin ──→ MapResource（地图几何）

2. 运行时（每 tick）：
   客户端 ──(Tick JSON)──→ 服务端
   服务端: apply_commands() → advance() → 生产 → 消费 → 人口增长
   服务端 ──(StateSnapshot bincode)──→ 客户端
   客户端: 更新顶点颜色 → 更新 HUD → 更新省份面板

3. 持久化：
   服务端每 30 tick 将 GameState 序列化为 JSON 存入 RocksDB
```

## 编译配置

### `.cargo/config.toml`

```toml
[build]
rustflags = ["-C", "link-arg=-fuse-ld=mold"]

[env]
CC = "clang"
CXX = "clang++"
```

- **clang**：用于编译 RocksDB 的 C++ 依赖（GCC 15 存在 `<cstdint>` 兼容性问题）。
- **mold**：超高速链接器，显著缩短增量编译的链接时间。

### 工作区 `Cargo.toml`

```toml
[profile.dev]
opt-level = 1                    # 开发构建启用基本优化
[profile.dev.package."*"]
opt-level = 3                    # 第三方依赖（尤其 Bevy）全优化
```

开发构建中对第三方依赖使用 `opt-level = 3`，确保 Bevy 等框架在调试时也有可接受的性能。
